❌ nu (nushell) — cargo install nu

Point of failure: Dockerfile.tools Step 6, proviso package install || true

Hypothesis: STALE IMAGE — when the tools image was built, nu was commented out in modern-linux-utils.conf. Proviso never attempted to install it. The || true hid nothing; the attempt was simply never made.

❌ hx (helix) — cargo install via git with wrong package name

Point of failure: Dockerfile.tools Step 6, _install_cargo_git() path in PackageInstall

Hypothesis: STALE IMAGE — tools image built with old config: method=cargo, repo="https://github.com/helix-editor/helix", package=hx. Proviso called cargo install --git https://github.com/helix-editor/helix hx --root /usr/local. The helix workspace has no package named hx (it is helix / helix-term). Install failed, || true swallowed the error.

Image's /tmp/provisions/ide.conf: helix { install { method = cargo, repo = "https://github.com/helix-editor/helix", package = hx } post_install = "hx --grammar fetch && hx --grammar build" } Current ide.conf: helix { install { method = cargo, package = helix-term } post_install = "hx --grammar fetch && hx --grammar build" }

❌ fish — dnf install fish never ran

Point of failure: Dockerfile.tools Step 6 — the shells { fish { ... } } block did not exist in ide.conf when the tools image was built

Hypothesis: STALE IMAGE — the fish provision was added to ide.conf after the last tools image build. Proviso never saw it. Fish is absent from the image. Dockerfile.devcontainer's --shell /usr/bin/fish in useradd does not verify the binary exists.

Image's /tmp/provisions/ide.conf — NO shells section. Image's /tmp/provisions/ide.conf — STILL HAS: xeyes, gtk3-demo, dbus-x11, jetbrainsTb, intellij, pycharm, libxcrypt-compat. All of these were removed from ide.conf in the last session but the tools image was never rebuilt.

🔧 Phase 5 — Fixes

🔑 Fix 1 (Root cause — resolves all 3 failures): Rebuild the tools image

All 3 failures share one root cause: the tools image predates the last session's conf file changes. A rebuild picks up: fish (dnf), helix-term (cargo), nu (cargo), removes envex ghost binary.

docker rmi proviso-dev-tools:latest && ./build.sh --docker

⚠️ This is the 30-60min cold build. Current conf files are correct — no changes needed before rebuilding.

🌲 Fix 2: Add tree to conf files

User added tree "while away" but it exists in neither conf files nor the image. Tree is in base UBI9 (no EPEL needed). Add to modern-linux-utils.conf under file-navigation or a new category, and add tree to setup.sh's _tools validation array.

file-navigation { ... tree { install { method = dnf, package = tree } } }

🐟 Fix 3 (Resilience): Give nushell its own Docker layer

Per .ai/CONTEXT.md: "nushell stays separate — its own Docker layer means its cache isn't busted when other crates change." Currently nu is in the same Step 6 as everything else. Heavy build (~15min). Add a dedicated step in Dockerfile.tools before Step 6:

RUN --mount=type=cache,target=/root/.cargo/registry \ --mount=type=cache,target=/root/.cargo/git \ --mount=type=cache,target=/root/.cargo/target \ cargo install --locked --root /usr/local nu || echo "nu build failed"

⚠️ Fix 4: Make Dockerfile.tools Step 6 failures visible

proviso package install || true silently swallows ALL failures. This is why the helix git-install failure was invisible for the entire lifetime of the broken image. Minimum fix: log exit code. Better: fail loudly but continue per-package.

proviso -vv -m /tmp/provisions/ package install; _rc=$?; echo "proviso exit: $_rc"; exit 0

🚀 Fix 5 (Performance): Restore parallel cargo installs

.ai/CONTEXT.md documents parallel cargo installs (xargs -P 6, per-crate CARGO_TARGET_DIR) that reduce build time from ~90min to ~30-35min. The current CargoProvider.install() is sequential. This is a documented architectural regression. Options: (a) implement parallelism in Dockerfile.tools directly using the original xargs approach alongside proviso, or (b) add a parallel install mode to CargoProvider.

🐳 Note: Docker socket warning is a Windows artifact

build.sh warns "docker socket not found" when run from Git Bash because readlink -f /var/run/docker.sock resolves nothing on Windows. Docker CLI IS installed in the tools image (Dockerfile.tools Step 2b). The socket mounts correctly in WSL2 at /var/run/docker.sock pointing to Docker Desktop's proxy.

Audit complete. Root cause confirmed. All 3 failures = stale tools image. All 31 ✅ items verified at runtime. Zero false positives. Next action: Fix 1 (rebuild), Fix 2 (tree), Fix 3 (nu layer).
